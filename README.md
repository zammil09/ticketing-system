# Blockchain Ticketing System — الخطوة ١: العقود الذكية

## هيكل المشروع
```
contracts/
  MockUSDC.sol          عملة مستقرة تجريبية (6 decimals)
  EventRegistry.sol      إدارة بيانات الفعاليات
  TicketNFT.sol           التذاكر كـ NFT (ERC-721)
  TicketMarketplace.sol   منطق الشراء الذري (دفع + سك في معاملة واحدة)
scripts/deploy.js        سكربت النشر (بالترتيب الصحيح + ربط الصلاحيات)
test/Ticketing.test.js   اختبارات تغطي: الشراء، Sold out، منع إعادة استخدام التذكرة
```

## التشغيل على جهازك (يحتاج اتصال إنترنت لتحميل الحزم)

```bash
npm install
npx hardhat compile
npx hardhat test
```

## النشر على Sepolia
1. انسخ `.env.example` إلى `.env` واملأ القيم.
2. احصل على ETH تجريبي من [Sepolia Faucet](https://sepoliafaucet.com).
3. نفّذ:
```bash
npm run deploy:sepolia
```
4. احفظ العناوين الأربعة التي تظهر في النهاية — راح تحتاجها بالـ Frontend (الخطوة القادمة).

## منطق الأمان المُطبَّق
- **العملية الذرية:** `buyTicket()` تنفّذ الدفع + السك بمعاملة واحدة. أي فشل يُلغي كل شيء تلقائياً (خاصية EVM) — مستحيل يدفع المستخدم بدون ما يستلم تذكرة.
- **ReentrancyGuard:** يمنع هجمات إعادة الدخول على دالة الشراء.
- **صلاحيات مقيّدة:** فقط عقد الـ Marketplace يقدر يسك تذاكر (`onlyMarketplace`) أو يحدّث عدد المبيعات.
- **منع إعادة استخدام التذكرة:** `checkIn()` يسجّل حالة الاستخدام على السلسلة.

## الخطوة القادمة
راح نبني الـ Frontend بـ React + Ethers.js يتصل بهذي العقود مباشرة (Connect Wallet, Buy Ticket, My Tickets).

## هيكل الواجهة
```
frontend/
  index.html          الصفحة الرئيسية (اختيار الدور: User / Event Owner)

```

## خطوات التشغيل
1. انشر العقود أولاً (الخطوة ١) واحصل على العناوين الأربعة.
2. افتح `frontend/js/config.js` وضع العناوين الحقيقية بدل placeholders.
3. شغّل الواجهة محلياً بأي سيرفر بسيط (لازم سيرفر وليس فتح الملف مباشرة، لأن المتصفحات تمنع بعض طلبات fetch من file://):
   ```bash
   cd frontend
   npx serve .
   # أو
   python3 -m http.server 8080
   ```
4. افتح المتصفح على `http://localhost:8080`، تأكد إن ميتاماسك مضبوطة على شبكة Sepolia.
5. اضغط "ربط المحفظة" → اختر الدور → جرّب الإنشاء/الشراء.

## ملاحظات تصميم مهمة (تفيدك بفصل التنفيذ بالبحث)
- **لا يوجد enumerable getter في العقود** — الواجهة تعتمد على قراءة أحداث `EventCreated` و `Transfer` من السلسلة (Event Logs) لبناء القوائم، بدل تخزين مصفوفات مكلفة غازياً داخل العقد. هذا نمط معماري شائع في dApps (Indexing via Logs).
- **عملية approve + buyTicket خطوتان منفصلتان** لأنها معيار ERC-20 القياسي — لازم توثقها بوضوح في شرح تدفق البيانات بالبحث.
- **التحقق من الشبكة تلقائياً** — الكود يطلب من ميتاماسك التبديل لشبكة Sepolia إذا كان المستخدم على شبكة مختلفة.
- في مشروع إنتاجي حقيقي يُفضّل استخدام backend + indexer (كما في مخطط الآركيتكتشر السابق) بدل قراءة كل الـ logs من المتصفح مباشرة، لأنها تصبح بطيئة مع كثرة الفعاليات.

## الخطوة القادمة
تجربة كاملة على Sepolia (نشر → ربط العناوين → إنشاء فعالية → شراء تذكرة → Check-in)، ثم توثيق النتائج بالصور لفصل التنفيذ.

# الخطوة ٣: بوابة الدخول (Check-in) + رموز QR

## الإضافات
- **`js/checkin.js`**: صفحة جديدة للمنظّم فيها مسح كاميرا حي (`html5-qrcode`) + إدخال يدوي لرقم التذكرة.
- **التحقق ثلاثي الطبقات** قبل قبول أي تذكرة:
  1. هل التوكن موجود فعلاً على السلسلة (`ownerOf`)؟
  2. هل الفعالية تتبع المنظّم الحالي المتصل؟
  3. هل استُخدمت من قبل (`ticketInfo.checkedIn`)؟
- عند القبول، يُستدعى `checkIn(tokenId)` مباشرة على العقد — الحالة تُسجَّل على السلسلة نفسها، وليس فقط بالواجهة، فلا يمكن التلاعب بها من طرف العميل.
- **رموز QR حقيقية**: كل تذكرة "صالحة" بصفحة "تذاكري" تعرض QR (مكتبة `qrcode`) يحمل `tokenId`، يُمسح مباشرة عبر كاميرا صفحة البوابة.

## ⚠️ ملاحظة أمنية مهمة (اذكرها في مناقشة البحث)
ترميز `tokenId` وحده بالـ QR بسيط لكنه غير آمن بالكامل — أي شخص يصوّر شاشة تذكرة غيره يقدر يحاول استخدامها (لن ينجح إلا مرة واحدة بسبب `checkedIn` على السلسلة، لكن "السباق" على من يمسحها أولاً يبقى وارد). التحسين المقترح لعمل مستقبلي: توقيع التذكرة بمفتاح خاص للمالك (EIP-712 signature) يتغيّر كل فترة، والتحقق من التوقيع وليس فقط الرقم.

## التشغيل
لازم HTTPS أو `localhost` لعمل الكاميرا (متطلب أمان من المتصفح). التشغيل المحلي بـ `python3 -m http.server` أو `npx serve` يفي بالغرض.

# الخطوة ٤: Backend + قاعدة بيانات (Node.js + Express + PostgreSQL)

## ليش أضفناه؟
الفرونت إند كان يقرأ كل شي مباشرة من البلوك تشين بكل مرة (بطيء + بدون بيانات مستخدمين حقيقية). الحل: **Indexer** يشتغل خلفياً، يسمع لأحداث العقود أول بأول، ويخزّنها بقاعدة بيانات PostgreSQL — فيصير الفرونت إند يقرأ من الداتابيس (سريع جداً) بدل ما يفحص السلسلة نفسها.

## هيكل Backend
```
backend/
  src/
    schema.sql        جداول: users, events, tickets, indexer_state
    db.js               اتصال PostgreSQL
    abi.js               الـ ABIs المطلوبة لقراءة الأحداث فقط
    indexer.js            ⭐ يفحص السلسلة كل 15 ثانية ويحدّث الجداول
    init-db.js             سكربت لإنشاء الجداول أول مرة
    server.js               Express app (نقطة البداية)
    routes/
      users.js        POST/GET بيانات المستخدم (id, name, email — نفس سكيتشك الأصلي)
      events.js         قراءة الفعاليات من الداتابيس (سريع)
      tickets.js          تذاكر المستخدم / تذاكر فعالية معيّنة
      dashboard.js          إحصائيات مجمّعة للمنظّم (إيرادات، حضور)
```

## مهم: العقود لا تزال المصدر الوحيد للحقيقة (Source of Truth)
- **الشراء / الإنشاء / تسجيل الدخول** لا تزال تتم مباشرة من الفرونت إند عبر MetaMask → العقد الذكي (كما بالخطوات السابقة). الباك إند **لا يكتب على السلسلة أبداً**.
- الباك إند فقط "يقرأ ويخزّن نسخة سريعة" مما حدث على السلسلة فعلاً — إذا تعطّل الباك إند بالكامل، الفرونت إند يرجع تلقائياً يقرأ من السلسلة مباشرة (fallback مبرمج بالفعل بـ `events.js` و`tickets.js`).
- هذا التصميم يُسمى "Read Model منفصل عن Write Model" — نقطة قوية تذكرها بمناقشة البحث.

## التشغيل المحلي

### 1) ثبّت PostgreSQL وأنشئ قاعدة بيانات
```bash
createdb ticketing_system
```

### 2) اضبط ملف `.env`
```bash
cd backend
cp .env.example .env
# عدّل DATABASE_URL وعناوين العقود ورابط RPC
```

### 3) ثبّت الحزم وأنشئ الجداول
```bash
npm install
npm run db:init
```

### 4) شغّل السيرفر (يبدأ الـ API + الـ Indexer معاً)
```bash
npm run dev
```
راح تشوف بالتيرمنال:
```
🚀 API شغّالة على http://localhost:4000
🔄 بدء تشغيل الـ Indexer — يفحص السلسلة كل 15 ثانية
[registry] فعالية جديدة #0: New Year's Party
```

### 5) الفرونت إند
`frontend/js/config.js` فيه `BACKEND_URL` مضبوط تلقائياً على `http://localhost:4000/api`. شغّل الفرونت إند بأي سيرفر محلي زي المعتاد.

## نقاط الـ API المتاحة
| Method | Endpoint | الوصف |
|---|---|---|
| POST | `/api/users` | إنشاء/تحديث ملف مستخدم |
| GET | `/api/users/:wallet` | جلب ملف مستخدم |
| GET | `/api/events` | كل الفعاليات |
| GET | `/api/events/organizer/:address` | فعاليات منظّم معيّن |
| GET | `/api/tickets/owner/:address` | تذاكر مستخدم معيّن |
| GET | `/api/tickets/event/:eventId` | كل تذاكر فعالية |
| GET | `/api/dashboard/organizer/:address` | إحصائيات مجمّعة |
